# datafusion-ffi-graal-test

GraalVM native image compatibility test for `datafusion-ffi-java`. Compiles the entire library — Java FFM bindings, Arrow vectors, and the Rust native library — into a single self-contained executable and runs a suite of integration tests against it.

## What is being tested

The test suite (`GraalNativeImageTest.java`) exercises multiple scenarios covering the most important types of FFI:
 - java.lang.foreign downcalls
 - java.lang.foreign upcalls
 - Arrow C Data JNI for zero-copy data transfer

## How it works

### Architecture

In normal JVM mode, `datafusion-ffi-java` loads a dynamic library (`.dylib`/`.so`) at runtime via `java.library.path`. For a GraalVM native image, the architecture is different:

1. **Static linking**: The Rust library (`libdatafusion_ffi_native.a`) is statically linked directly into the native image binary at build time. There is no separate `.dylib` to load at runtime.

2. **Symbol discovery**: At runtime, `NativeLoader` detects it's running in a native image (via `org.graalvm.nativeimage.imagecode` system property) and calls `SymbolLookup.libraryLookup()` on the executable itself. Since the Rust symbols are statically linked into the binary, `dlsym` finds them directly.

3. **Build-time metadata**: GraalVM's native image compiler uses ahead-of-time (AOT) analysis and needs explicit metadata for features it can't statically determine: reflection targets, JNI access, resource files, and FFM function descriptors.

### Running the tests

There are two execution modes:

**JVM mode** (uses the dynamic library, no GraalVM needed):
```bash
./gradlew :datafusion-ffi-graal-test:run
```

**Native image mode** (compiles everything into a single binary):
```bash
./datafusion-ffi-graal-test/scripts/run-native-test.sh
```

The script detects `JAVA_HOME` (for Gradle) and `GRAALVM_HOME` (for `native-image`), builds the native binary, and runs it. Use `--skip-build` to re-run an already-built binary.

## Dependencies

### 1. JDK 21+ (for Gradle)

The Gradle daemon requires JDK 21 to build the project. Set `JAVA_HOME` to a JDK 21 installation.

- **macOS (Homebrew)**: `brew install --cask temurin@21`
- **macOS (manual)**: Download from [Adoptium](https://adoptium.net/temurin/releases/?version=21)
- **Linux**: `sudo apt install temurin-21-jdk` or equivalent

### 2. GraalVM JDK 25+ (for native-image)

The native image build requires Oracle GraalVM JDK 25 or later. This provides the `native-image` tool and supports the Java FFM API features used by the library (`SharedArenaSupport`, FFM upcalls on macOS ARM64).

- **macOS (manual)**: Download from [Oracle GraalVM Downloads](https://www.graalvm.org/downloads/). Install to `~/Library/Java/JavaVirtualMachines/` so the build script auto-detects it.
- **Linux**: Download the appropriate tarball from the same page.

Set `GRAALVM_HOME` to the GraalVM installation root (the directory containing `bin/native-image`), or place it in a standard location for auto-detection.

### 3. Rust toolchain

The Rust native library must be built before the native image can be compiled. The `Cargo.toml` must include `crate-type = ["cdylib", "staticlib"]` — `staticlib` produces the `.a` archive that gets linked into the native image.

- **Install**: https://rustup.rs/

Build the library (the Gradle build does this automatically):
```bash
cd datafusion-ffi-native && cargo build
```

### 4. Apache Arrow Java 19.0.0-SNAPSHOT

The `build.gradle` forces all Arrow dependencies to `19.0.0-SNAPSHOT` via `resolutionStrategy.force(...)`. This is required because Arrow C Data JNI in native image depends on a [FindClass JNI fix](https://github.com/apache/arrow-java/pull/399) that changes class name format from `Ljava/lang/Object;` to `java/lang/Object` — the former causes `FindClass` to fail in GraalVM's SubstrateVM JNI implementation.

This snapshot must be installed in the local Maven repository (`~/.m2/repository`):
```bash
git clone https://github.com/apache/arrow-java.git
cd arrow-java
mvn install -DskipTests -pl vector,c,format,memory/memory-core,memory/memory-unsafe
```

## GraalVM native image configuration

GraalVM's native image compiler performs closed-world analysis at build time. Unlike the JVM, it cannot discover classes, methods, or resources at runtime unless they are explicitly registered. Two configuration mechanisms work together to make `datafusion-ffi-java` compatible:

### reachability-metadata.json

`src/main/resources/META-INF/native-image/graaltest/reachability-metadata.json`

This JSON file declares **what the native image compiler cannot discover through static analysis**. It was initially generated by running the GraalVM tracing agent (`-Pagent`) and then manually curated. It contains four sections:

- **`reflection`**: Classes and methods accessed via reflection at runtime. This includes:
  - Arrow memory internals (`DefaultAllocationManagerFactory`, `DirectByteBuffer`) that Arrow discovers via `Class.forName()` and reflection
  - Arrow schema types (`Field`, `Schema`, `ArrowType`) whose fields are accessed by Jackson for JSON serialization
  - Handle classes (`CatalogProviderHandle`, `SchemaProviderHandle`, `TableProviderHandle`, `ExecutionPlanHandle`, `RecordBatchReaderHandle`) whose callback methods are invoked via `MethodHandles.lookup().findVirtual()` — GraalVM needs reflection metadata to make these methods available to the MethodHandle lookup
  - Logging framework configurators (Logback)

- **`resources`**: Classpath resources loaded at runtime via `getResourceAsStream()`. Includes SPI service files, the Arrow C Data JNI native library (`arrow_cdata_jni/*.dylib`), and logging configuration.

- **`jni`**: Classes and methods called from native code via JNI. The Arrow C Data JNI library (`libarrow_cdata_jni`) uses JNI to call back into Java (`ArrayStreamExporter`, `PrivateData.close()`). Without these entries, JNI calls from the Arrow C library would fail with `NoSuchMethodError`.

- **`bundles`**: Resource bundles (currently empty).

### ForeignRegistrationFeature.java

`src/main/java/org/apache/arrow/datafusion/graaltest/ForeignRegistrationFeature.java`

This is a GraalVM [Feature](https://www.graalvm.org/latest/reference-manual/native-image/overview/BuildConfiguration/#features) class that runs at **image build time** and registers every `FunctionDescriptor` signature used by the library for FFM downcalls and upcalls.

In a normal JVM, `Linker.downcallHandle()` and `Linker.upcallStub()` generate calling convention adapters on-the-fly from `FunctionDescriptor` signatures. In a native image, these adapters must be generated at build time because runtime code generation is not available. `RuntimeForeignAccess.registerForDowncall()` and `registerForUpcall()` tell the native image compiler which signatures to pre-generate.

**Downcalls** (Java -> Rust): Every unique `FunctionDescriptor` signature used in `NativeUtil.downcall()` calls across all `*Ffi` classes must be registered. These use only simple `ValueLayout` types (`ADDRESS`, `JAVA_INT`, `JAVA_LONG`).

**Upcalls** (Rust -> Java): Every unique `FunctionDescriptor` signature used in `Linker.upcallStub()` calls across all `*Handle` classes must be registered. These are more complex because many callbacks return `abi_stable` struct types (`RVec`, `RString`, `ROption`, `FFIResult`, `FfiPoll`) which require `StructLayout` descriptors.

The struct layout sizes in `ForeignRegistrationFeature` are **hardcoded** because the Rust native library is not loaded at image build time (the `org.apache.arrow.datafusion` package is configured with `--initialize-at-run-time`). The hardcoded values are derived from the known ABI-stable struct layouts and are validated at test time by `FfiSizeValidationTest` in the main module.

The Feature is activated in `build.gradle` via:
```
--features=org.apache.arrow.datafusion.graaltest.ForeignRegistrationFeature
```

### Why both are needed

The two mechanisms serve complementary purposes and neither can replace the other:

| Concern | reachability-metadata.json | ForeignRegistrationFeature |
|---------|---------------------------|---------------------------|
| Reflection (Class.forName, field access) | Yes | No |
| JNI callbacks (Arrow C Data) | Yes | No |
| Resource loading (SPI, native libs) | Yes | No |
| FFM downcall stubs (Java -> Rust) | No | Yes |
| FFM upcall stubs (Rust -> Java) | No | Yes |

## Current limitations and workarounds

### 1. Arrow Java 19.0.0-SNAPSHOT required

**Problem**: Arrow C Data JNI fails in GraalVM native image with `FindClass` errors because the JNI code uses descriptor-format class names (`Ljava/lang/Object;`) instead of internal-format names (`java/lang/Object`).

**Workaround**: Force all Arrow dependencies to `19.0.0-SNAPSHOT` which includes the fix (PR [apache/arrow-java#399](https://github.com/apache/arrow-java/pull/399)).

**Resolution**: Once Arrow Java 19.0.0 is released as a stable version, update the `arrowVersion` in `build.gradle` and remove the `resolutionStrategy.force(...)` block.

### 2. Hardcoded struct sizes in ForeignRegistrationFeature

**Problem**: The Feature class hardcodes byte sizes for Rust FFI structs (e.g., `FFI_SchemaProvider = 264 bytes`, `FFI_SessionRef = 264 bytes`) because the native library is not available at image build time. If upstream `datafusion-ffi` changes struct layouts, the hardcoded values silently become wrong.

**Workaround**: Sizes are documented with their derivation (e.g., `owner_name(40) + 5 fn ptrs(40) + logical_codec(144) + 5 fields(40) = 264`). The main module's `FfiSizeValidationTest` validates Java layout sizes against Rust at JVM test time, so mismatches are caught by `./gradlew :datafusion-ffi-java:test` before the native image is built.

**Resolution**: GraalVM could support initializing specific classes at build time to query sizes from the native library, but this is not currently feasible because the FFI classes have complex static initializers that depend on the full Rust runtime. A future improvement could extract size constants into a separate class with no native dependencies.

### 3. Reachability metadata must be manually maintained

**Problem**: When new reflection targets, JNI methods, or resources are added to `datafusion-ffi-java` or its dependencies, the `reachability-metadata.json` must be updated or the native image will fail at runtime with `ClassNotFoundException`, `NoSuchMethodError`, or missing resource errors.

**Workaround**: Run the GraalVM tracing agent to discover new metadata:
```bash
# Run with agent (requires GraalVM JDK set as JAVA_HOME)
./gradlew :datafusion-ffi-graal-test:run -Pagent

# Copy agent output to metadata file
./gradlew :datafusion-ffi-graal-test:metadataCopy
```
Then manually review and merge the generated metadata with the existing file.

**Resolution**: The Arrow and DataFusion libraries could ship their own GraalVM reachability metadata (via `META-INF/native-image/` in their JARs), which would make per-project metadata files unnecessary.

### 4. ForeignRegistrationFeature must be manually maintained

**Problem**: When new FFM function descriptors are added to `datafusion-ffi-java` (new downcall or upcall signatures), the Feature class must be manually updated with the new signatures. Missing registrations cause `IllegalArgumentException` at runtime when `Linker.downcallHandle()` or `Linker.upcallStub()` is called.

**Workaround**: When adding new FFI functions, add the corresponding `FunctionDescriptor` registration to `ForeignRegistrationFeature`. Downcall signatures only use simple `ValueLayout` types. Upcall signatures may require `StructLayout` construction — follow the patterns in the existing `registerUpcalls()` method.

**Resolution**: `datafusion-ffi-java` could ship a Feature class in its own JAR (auto-discovered via `META-INF/native-image/`), eliminating the need for consumers to maintain their own. This is blocked on finding a clean way to express struct layout sizes without native library access at build time.

### 5. macOS symbol export workaround

**Problem**: On macOS, statically linked symbols are not exported by default. The FFM `SymbolLookup.libraryLookup()` uses `dlsym()` to find symbols, which only searches exported symbols.

**Workaround**: The `build.gradle` runs `nm -g` on the static library at build time to discover all `_datafusion_*` global symbols, then passes individual `-Wl,-exported_symbol,<sym>` flags to the linker. On Linux, the simpler `--export-dynamic` flag exports all symbols.

**Resolution**: This is inherent to macOS's linker behavior and is unlikely to change. The workaround is robust and automatic.

### 6. `-H:+SharedArenaSupport` is experimental

**Problem**: The `SharedArenaSupport` GraalVM option (required for `Arena.ofShared()` used by catalog registration) is currently experimental and produces a build warning.

**Workaround**: The option is passed in `build.gradle` build args. The warning is harmless.

**Resolution**: This will become stable in a future GraalVM release and the warning will disappear.
