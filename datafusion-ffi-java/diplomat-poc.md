# Diplomat POC Plan for datafusion-ffi-java

This document describes a proof-of-concept to auto-generate Java FFI bindings for DataFusion using [Diplomat](https://rust-diplomat.github.io/diplomat/), replacing hand-written `*Ffi.java` and `*Handle.java` classes and eliminating the hand-written Rust FFI boilerplate entirely.

## Motivation

The current architecture requires two parallel hand-written layers for every FFI operation:

1. **Rust**: `#[no_mangle] extern "C"` functions with manual pointer casting, error handling, and `unsafe` blocks (e.g., `session_context.rs` is ~800 lines for 15 functions)
2. **Java**: `*Ffi.java` classes with `MethodHandle` constants, `FunctionDescriptor` declarations, and `invokeExact` calls (e.g., `SessionContextFfi.java` is ~640 lines), plus `*Handle.java` classes with upcall stubs for Java→Rust callbacks

With Diplomat:
- The **Rust side** becomes clean `#[diplomat::bridge]` modules with idiomatic Rust methods and traits. The Diplomat proc-macro generates all the `#[no_mangle] extern "C"` glue, vtables, and callback plumbing.
- The **Java side** is entirely auto-generated by `diplomat-tool`. No hand-written `MethodHandle` constants, no `FunctionDescriptor` boilerplate, no manual upcall stub creation.

## Scope

The POC targets **SessionContext** as the representative type, covering ALL of its methods — including the callback-based ones (`registerCatalog`, `registerUdf`, `registerListingTable`). The goal is a fully Diplomat-generated FFI layer with no hand-written `*Ffi.java` or `*Handle.java` residual.

### All SessionContext methods go through Diplomat

| Method | Diplomat mechanism | Java backend feature needed |
|--------|-------------------|---------------------------|
| `new()` | Opaque constructor | Already supported |
| `newWithConfig()` | Opaque constructor with struct param | Structs |
| `newWithConfigRt()` | Constructor with opaque + struct params | Structs |
| `sql()` | Fallible return: `Result<Box<DataFrame>, Box<DfError>>` | Fallible returns |
| `sessionId()` | `DiplomatWrite` string return | Already supported |
| `sessionStartTime()` | Primitive return (`i64`) | Already supported |
| `state()` | Returns `Box<DfSessionState>` opaque | Already supported |
| `registerTable()` | Takes opaque `DfArrowBatch` wrapping Arrow FFI | Opaques as params |
| `registerCatalog()` | Takes `impl DfCatalogTrait` | **Traits** |
| `registerUdf()` | Takes `impl DfScalarUdfTrait` | **Traits** |
| `registerListingTable()` | Takes struct + `impl DfFileFormatTrait` | **Traits** + structs |
| `parseSqlExpr()` | Takes string + opaque schema, returns `&[u8]` | Slices |
| `close()` | Auto-generated destructor | Already supported |

### What changes for Handle classes

The current `*Handle.java` classes (CatalogProviderHandle, SchemaProviderHandle, TableProviderHandle, ExecutionPlanHandle, RecordBatchReaderHandle, FileFormatHandle, FileSourceHandle, FileOpenerHandle, ScalarUdfHandle) are manually-written upcall bridges that create FFM `upcallStub` function pointers, populate FFI struct fields, and manage Arena lifecycles.

With Diplomat traits, all of this is auto-generated:
- Diplomat generates Java interfaces matching each trait
- Diplomat generates wrapper classes that create the vtable with upcall stubs
- The proc-macro generates the Rust-side trait implementation that dispatches through the vtable

The hand-written `*Handle.java` classes are replaced by Diplomat-generated trait wrappers.

## Architecture

### Current architecture

```
SessionContext.java (public API)
    └── SessionContextFfi.java (hand-written, 640 lines)
            ├── MethodHandle downcalls → session_context.rs (hand-written, 800 lines)
            └── *Handle.java classes (hand-written, ~500 lines each)
                    └── upcallStub → Rust callback dispatch
```

### Proposed architecture (fully generated)

```
SessionContext.java (public API, simplified)
    └── DfSessionContext.java (DIPLOMAT-GENERATED)
            ├── MethodHandle downcalls → bridge.rs (#[diplomat::bridge], ~200 lines)
            └── DfCatalogTrait.java (DIPLOMAT-GENERATED trait interface + wrapper)
                    └── upcallStub → Diplomat-generated vtable dispatch
```

No `SessionContextFfi.java`. No `*Handle.java`. No `session_context.rs`. The entire FFI layer is Diplomat-generated from `bridge.rs`.

## How Diplomat Traits Replace Handle Classes

Diplomat's [trait support](https://rust-diplomat.github.io/diplomat/traits.html) enables foreign-language implementations of Rust traits. Internally, traits are vtables with function pointers — the same pattern the current `*Handle.java` classes implement manually.

### Current pattern (hand-written)

```
Java: CatalogProviderHandle
  1. Creates MethodHandle for each callback method
  2. Creates upcallStub (function pointer) for each via Linker.upcallStub()
  3. Allocates FFI_CatalogProvider struct in Arena
  4. Populates struct fields with upcallStub pointers
  5. Implements copyStructTo() to transfer to Rust

Rust: session_context.rs
  1. Reads FFI_CatalogProvider from pointer
  2. Converts via TryFrom to ForeignCatalogProvider
  3. Registers with DataFusion
```

### Diplomat pattern (auto-generated)

```
Rust bridge.rs:
  pub trait DfCatalogTrait {
      fn schema_names(&self, write: &mut DiplomatWrite);
      fn schema(&self, name: &DiplomatStr) -> Option<Box<DfSchemaProvider>>;
  }

Diplomat proc-macro generates:
  - C vtable struct with function pointers for each method
  - impl DfCatalogTrait for the vtable struct (dispatches through function pointers)

diplomat-tool generates (Java):
  - public interface DfCatalogTrait { ... }
  - internal DfCatalogTrait_Wrapper (creates upcall stubs, populates vtable)
```

The Java user implements `DfCatalogTrait`, and the generated wrapper handles all the FFI plumbing — exactly what `CatalogProviderHandle` does today, but without hand-writing it.

### Trait method capabilities

Based on analysis of Diplomat's HIR and feature tests, trait methods support:

| Pattern | Supported | Notes |
|---------|-----------|-------|
| Primitive params/returns | Yes | `fn method(&self, x: i32) -> i32` |
| Struct params/returns | Yes | `fn method(&self, s: MyStruct) -> MyStruct` |
| `&DiplomatStr` params | Yes | `fn method(&self, name: &DiplomatStr)` |
| `DiplomatWrite` returns | Yes | `fn method(&self, write: &mut DiplomatWrite)` |
| `&OpaqueType` params | Yes | `fn method(&self, other: &MyOpaque)` |
| `Box<OpaqueType>` returns | Yes | `fn method(&self) -> Box<MyOpaque>` |
| `Option<Box<OpaqueType>>` | Yes | `fn method(&self) -> Option<Box<MyOpaque>>` |
| `Result<T, E>` returns | Yes (at HIR level) | Disabled for Kotlin; needs Java impl |
| `impl OtherTrait` params | Yes | For passing nested traits |
| `&[u8]` slice params | Yes | For byte/primitive slices |
| Returning `impl Trait` | **No** | Must return `Box<Opaque>` instead |

The key limitation — **trait methods cannot return other traits** — is handled by returning `Box<OpaqueType>` where the opaque wraps another trait implementation (see "Provider chain" below).

### The provider chain pattern

DataFusion's `CatalogProvider → SchemaProvider → TableProvider → ExecutionPlan → RecordBatchReader` chain requires each level to return the next. Since Diplomat traits can't return `impl Trait`, we use opaque wrappers:

```rust
#[diplomat::bridge]
mod ffi {
    // Each level is both an opaque type AND has a trait for Java implementation

    pub trait DfCatalogTrait {
        fn schema_names(&self, write: &mut DiplomatWrite);
        fn schema(&self, name: &DiplomatStr) -> Option<Box<DfSchemaProvider>>;
    }

    #[diplomat::opaque]
    pub struct DfSchemaProvider(Box<dyn SchemaProviderBridge>);

    // Java creates DfSchemaProvider by wrapping its DfSchemaTrait impl
    impl DfSchemaProvider {
        pub fn new(t: impl DfSchemaTrait) -> Box<DfSchemaProvider> {
            Box::new(DfSchemaProvider(Box::new(TraitWrapper(t))))
        }
    }

    pub trait DfSchemaTrait {
        fn table_names(&self, write: &mut DiplomatWrite);
        fn table(&self, name: &DiplomatStr) -> Option<Box<DfTableProvider>>;
    }

    // ... same pattern for TableProvider, ExecutionPlan, RecordBatchReader
}
```

When Rust calls `catalog.schema("my_schema")`:
1. The Diplomat vtable dispatches to the Java callback
2. Java creates a `DfSchemaProvider` by calling `DfSchemaProvider.new_(mySchemaImpl)` — a re-entrant downcall that creates the Rust opaque
3. Java returns the opaque pointer from the callback
4. Rust receives `Box<DfSchemaProvider>` and uses it

This re-entrant FFI pattern (downcall during upcall) is supported by both FFM and JNA.

## Modeling Arrow C Data Interface in Diplomat

The Arrow C Data Interface (FFI_ArrowSchema, FFI_ArrowArray) uses raw pointer passing with release callbacks. This doesn't map directly to any Diplomat type, but can be modeled via opaque wrappers with `usize` pointer addresses.

### Approach: Opaque wrappers with address-based factories

```rust
#[diplomat::bridge]
mod ffi {
    #[diplomat::opaque]
    pub struct DfArrowBatch {
        schema: FFI_ArrowSchema,
        array: FFI_ArrowArray,
    }

    impl DfArrowBatch {
        /// Create from raw FFI_ArrowSchema/FFI_ArrowArray addresses.
        /// Java calls this after Data.exportVectorSchemaRoot().
        pub fn from_addresses(schema_addr: usize, array_addr: usize) -> Box<DfArrowBatch> {
            unsafe {
                Box::new(DfArrowBatch {
                    schema: std::ptr::read(schema_addr as *const FFI_ArrowSchema),
                    array: std::ptr::read(array_addr as *const FFI_ArrowArray),
                })
            }
        }
    }

    impl DfSessionContext {
        pub fn register_table(
            &self,
            name: &DiplomatStr,
            batch: &DfArrowBatch,
        ) -> Result<(), Box<DfError>> {
            // Import Arrow data from the FFI structs, create MemTable, register
        }
    }
}
```

From Java:
```java
// Export VectorSchemaRoot to Arrow C Data Interface
ArrowSchema ffiSchema = ArrowSchema.allocateNew(allocator);
ArrowArray ffiArray = ArrowArray.allocateNew(allocator);
Data.exportVectorSchemaRoot(allocator, root, null, ffiArray, ffiSchema);

// Pass raw addresses through Diplomat opaques
DfArrowBatch batch = DfArrowBatch.fromAddresses(
    ffiSchema.memoryAddress(), ffiArray.memoryAddress());
ctx.registerTable(name, batch);
```

The same pattern works for `parseSqlExpr()` (schema address in, byte slice out) and for trait methods that need Arrow schemas (e.g., `TableProvider.schema()` writes an Arrow schema to a provided address).

For **returning** Arrow data from trait callbacks (e.g., `ExecutionPlan.execute()` returning record batches), the trait method can return a `Box<DfArrowBatch>` created by the Java callback from exported Arrow data.

## Diplomat Java Backend: Features to Add

The local Diplomat fork's Java backend needs significant additions. Listed in dependency order.

### 1. Fallible returns (`Result<T, E>`) — CRITICAL, BLOCKING

**Current state:** `is_method_supported()` returns `false` for `ReturnType::Fallible`. Nearly every SessionContext method needs this.

**What to implement in `tool/src/java/mod.rs`:**
- Handle `ReturnType::Fallible(ok_type, err_type)` in `get_return_layout()`, `gen_return_type_java()`, and `gen_method()`
- The generated C function returns a `DiplomatResult` struct by value. In Java FFM:
  1. Define a `StructLayout` for `DiplomatResult` (discriminant byte + union of ok/err)
  2. Use `FunctionDescriptor.of(resultLayout, ...)` for the downcall
  3. Read the `is_ok` discriminant from the returned `MemorySegment`
  4. On success: extract the ok value (pointer for opaques, primitive for primitives)
  5. On error: extract the error opaque, call its stringifier, throw `RuntimeException`
- Map the Java return type to the ok type (error becomes an exception)

**Reference:** Kotlin backend's `ResultUnitUnit` / `ResultPointerPointer` JNA Structures. The Java FFM approach uses `MemorySegment` with `StructLayout` instead.

**Estimated effort:** Medium. This is the highest-priority addition.

### 2. Structs — IMPORTANT

**Current state:** `TypeDef::Struct` is completely skipped in `run()`. Needed for config entries, listing table options.

**What to implement:**
- Add `TypeDef::Struct(s)` case in `run()` loop
- Generate Java classes with public fields matching struct fields
- Handle struct parameters (marshal Java fields to C struct layout)
- Handle struct returns (read C struct into Java fields)
- Create `Struct.java.jinja` template

**Estimated effort:** Medium.

### 3. Traits — CRITICAL for eliminating Handle classes

**Current state:** `attr_support().traits = false`, `callbacks = false`. No trait or callback support in the Java backend.

**What to implement (port from Kotlin backend to FFM):**

The Kotlin backend generates traits via JNA's `Callback` interface + `Structure` for vtables. The Java FFM equivalent:

1. **Generate a Java interface** for each Diplomat trait (matching the Kotlin `interface TesterTrait` pattern)

2. **Generate a vtable wrapper class** (replacing JNA Structures with FFM StructLayout):
   - `StructLayout` matching the C vtable struct (destructor + method function pointers)
   - For each trait method, create a `MethodHandle` pointing to a Java callback method
   - Use `Linker.upcallStub()` to create native function pointers from the MethodHandles
   - Populate the vtable struct fields with the upcall stubs

3. **Generate `fromTraitObj()` factory** that takes a Java interface impl and creates the vtable wrapper:
   - Creates callback closures capturing the trait object
   - Binds each closure to an upcall stub
   - Returns the populated vtable struct as a `MemorySegment`

4. **JVM cookie mechanism** — The Kotlin backend uses JNI GlobalRef to prevent GC of the Java object while Rust holds a pointer. For FFM, two options:
   - **Option A (simpler):** Use `Arena.ofShared()` — the Arena keeps the upcall stubs alive, and we store the wrapper object reference in a static map keyed by the data pointer. This matches the current `*Handle.java` pattern.
   - **Option B:** Port the JVM cookie mechanism (JNI GlobalRef) to FFM. More complex but what Kotlin does.

**Key files to modify in the Diplomat fork:**
- `tool/src/java/mod.rs` — Add trait/callback iteration, vtable generation logic
- `tool/templates/java/Trait.java.jinja` — New template (interface + wrapper)
- `tool/src/java/formatter.rs` — Trait type name formatting

**Estimated effort:** Large. This is the most complex addition but directly replaces all `*Handle.java` classes.

### 4. Callbacks (`impl Fn(...)`) — MEDIUM PRIORITY

**Current state:** `attr_support().callbacks = false`.

Callbacks are simpler than traits (single function, no vtable). Needed for simpler patterns like `impl Fn(i32) -> i32`. Implementation is a subset of trait support.

**Estimated effort:** Small-Medium (once traits work, callbacks are easy).

### 5. Enums as Java enum classes — NICE TO HAVE

**Current state:** Enums are passed as raw `int`. Works functionally but isn't idiomatic Java.

**What to implement:**
- Generate `public enum MyEnum { VARIANT_A, VARIANT_B, ... }` with an `int` value field
- Generate `fromInt()` / `toInt()` conversion methods
- Use the enum type instead of `int` in method signatures

**Estimated effort:** Medium.

### 6. Slices (primitive and string) — IMPORTANT

**Current state:** Only `&DiplomatStr` works. Needed for `&[usize]` projections, `&[DiplomatStr]` config entries.

**What to implement:**
- `&[u8]`, `&[i32]`, `&[usize]` etc. — allocate in Arena, pass as (pointer, length) pair
- `&[DiplomatStr]` (slice of strings) — allocate array of `DiplomatStringView` structs
- `DiplomatOption<T>` for optional primitives

**Estimated effort:** Small-Medium.

### 7. Library loading customization — IMPORTANT

**Current state:** Generated Java uses `System.loadLibrary()` + `SymbolLookup.loaderLookup()`.

**What to change:** Modify `Lib.java.jinja` and `Opaque.java.jinja` to delegate library loading to a `DiplomatLib.lookup()` method. The generated `DiplomatLib.java` either calls the existing `NativeLoader.get()` or is manually replaced with one that does.

**Estimated effort:** Small.

## Implementation Phases

### Phase 1: Diplomat Java backend — fallible returns + structs

Add `Result<T, E>` and struct support to the Java backend. This unblocks the core SessionContext methods.

**Validate:** Snapshot tests in the Diplomat fork + the Diplomat example project exercises these features.

### Phase 2: Minimal bridge — lifecycle + SQL + metadata

Create `bridge.rs` with `DfSessionContext` (new, sql, sessionId, sessionStartTimeMillis, close) and `DfDataFrame` (lifecycle only). Wire into Gradle build. Verify generated Java compiles and works.

**Validate:** New integration test that creates a SessionContext via Diplomat, runs SQL, checks sessionId.

### Phase 3: Arrow data wrappers

Add `DfArrowBatch` and `DfArrowSchema` opaques with address-based factories. Implement `registerTable()` and `parseSqlExpr()`.

**Validate:** Existing `IntegrationTest` test cases for registerTable pass through the Diplomat path.

### Phase 4: Diplomat Java backend — traits

Port trait support from the Kotlin backend to Java FFM. This is the largest single effort.

**Validate:** Diplomat feature tests for traits compile and pass with the Java backend.

### Phase 5: Catalog/schema/table provider traits

Add `DfCatalogTrait`, `DfSchemaTrait`, `DfTableTrait`, `DfExecutionPlanTrait`, `DfRecordBatchReaderTrait` to `bridge.rs`. Implement `registerCatalog()`.

**Validate:** Existing `IntegrationTest` test cases for custom catalogs pass through the Diplomat path.

### Phase 6: UDF + Listing table traits

Add `DfScalarUdfTrait`, `DfFileFormatTrait`, `DfFileSourceTrait`, `DfFileOpenerTrait`. Implement `registerUdf()` and `registerListingTable()`.

**Validate:** All existing integration tests pass.

### Phase 7: Remove hand-written FFI

Delete `SessionContextFfi.java`, all `*Handle.java` classes, and the corresponding Rust modules. Update `SessionContext.java` to delegate purely to the generated `DfSessionContext`.

**Validate:** Full test suite passes. Line-count comparison shows the reduction.

## Risks and Mitigations

### Risk 1: Trait complexity in the Java FFM backend

Implementing Diplomat traits for Java FFM is the hardest part. The Kotlin backend uses JNA's magic struct marshalling; FFM requires explicit `StructLayout` + `VarHandle` + `upcallStub()`.

**Mitigation:** Start with the simplest trait (`DfCatalogTrait` — two methods, primitive returns) and iterate. The pattern is repetitive once established.

### Risk 2: Re-entrant FFI (downcall during upcall)

The provider chain requires Java callbacks to make Diplomat downcalls (e.g., `DfSchemaProvider.new_(myImpl)`) while inside a Rust→Java upcall. If FFM doesn't handle re-entrancy, this deadlocks.

**Mitigation:** FFM supports re-entrant calls by design (each call gets its own stack frame). Validate with a minimal test before building the full chain.

### Risk 3: Arrow data address passing is fragile

Passing `usize` addresses for FFI_ArrowSchema/FFI_ArrowArray relies on the Java-side Arrow objects living long enough. If they're GC'd before Rust reads the data, we get a use-after-free.

**Mitigation:** The Diplomat opaque factory (`DfArrowBatch.fromAddresses()`) immediately copies the data via `ptr::read`, so the Java objects only need to live through the factory call. Use `try-with-resources` in Java to ensure this.

### Risk 4: DiplomatResult struct layout varies by type

`DiplomatResult<*mut T, *mut E>` and `DiplomatResult<i32, ()>` have different sizes. The Java backend must compute layouts dynamically based on the ok/err types.

**Mitigation:** Generate specific `StructLayout` constants per result type in the Java code, similar to how the Kotlin backend generates specific `ResultXxxYyy` JNA Structures.

### Risk 5: Public API interface mismatch

The generated `DfCatalogTrait` interface may not match the current `CatalogProvider` interface exactly (e.g., `schemaNames()` returns `List<String>` currently but would use `DiplomatWrite` in Diplomat).

**Mitigation:** The public `CatalogProvider.java` interface stays unchanged. An adapter in `SessionContext.java` converts between the public interface and the generated Diplomat trait interface. Over time, the public interface can evolve to match the generated one more closely.

### Risk 6: Diplomat Java backend maturity

The Java backend is early-stage. Adding traits, structs, and fallible returns is substantial work that may surface Diplomat core issues.

**Mitigation:** The phased approach lets us validate each feature independently. Phases 1-3 (no traits) already prove significant value. Traits (Phases 4-6) can be deferred if they prove too difficult without blocking the core POC.

## Success Criteria

The POC is successful if:

1. `./gradlew :datafusion-ffi-java:test` passes all existing tests
2. `SessionContextFfi.java` and all `*Handle.java` classes are deleted
3. `session_context.rs` and all Rust callback modules are deleted
4. The Diplomat `bridge.rs` is under 300 lines total
5. No regression in the public `SessionContext` API — all existing callers work unchanged
6. The number of hand-written FFI lines (Java + Rust combined) is reduced by >80%
